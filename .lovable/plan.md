I found the problem area. The Concierge currently mixes two different sources of truth:

1. The customized first-login Welcome Panel stored in the database (`onboarding_flow_config`).
2. The route/tone/language default opener generated by `greetingForContext()` in `conciergeGreeting.ts`.

Language/tone changes can still trigger the route-based default opener logic, while the real first-login custom message is only injected by an event from `/trade`. That is why only the lower action bubbles may change, or the panel can fall back to unrelated/old timeline content instead of preserving your customized welcome.

Plan:

1. Make the customized Welcome Panel the single source of truth for first-login/replay welcome
   - Add a small shared helper module for loading and rendering the onboarding welcome config.
   - It will fetch `onboarding_flow_config.default`, apply `{first_name}`, `{first_name_comma}`, and `{concierge_name}`, and return the exact greeting + buttons configured in the admin Welcome Panel.
   - Use this helper from both `TradeDashboard.tsx` and `replayWelcome.ts` so they cannot drift.

2. Give first-login welcome messages an explicit identity in the AI Concierge timeline
   - Mark the injected welcome timeline item as a `welcome`/`onboarding` message when it comes from the first-login flow.
   - Stop treating it like a route-generated default opener.
   - This prevents language/tone/path effects from overwriting it with `greetingForContext()`.

3. Fix language switching behavior
   - When the user changes language while the customized welcome is showing, preserve the exact customized greeting text from the Welcome Panel.
   - Do not resurrect any old built-in welcome copy.
   - Do not replace it with route-stage copy.
   - Keep the same buttons and same action behavior.
   - The selected language will continue to control AI replies via `toneSystemNote(tone, lang)`, and the picker UI will update labels/descriptions as it already does.

4. Ensure first-login flow behaves the same in every language
   - `Start Quick Tour` still starts the same tour.
   - `Start from a brief` still opens the same brief wizard.
   - `Rename {concierge_name}` still opens the rename panel.
   - Switching language before using those buttons will not change the flow or reset the panel.
   - The Quick Tour itself remains driven by the admin tour-step database rows, including the toolkit step.

5. Remove old-version fallback paths
   - Replace duplicated hardcoded default welcome strings in `TradeDashboard.tsx` and `replayWelcome.ts` with the shared helper fallback.
   - Update the Concierge reset/new-conversation behavior so it intentionally resets to a route contextual opener only when starting a normal new chat, not when the onboarding welcome is currently active unless explicitly intended.
   - Make sure the old assignment/project prompt cannot appear as the welcome state; assignment UI remains isolated to tearsheet proposal approval only.

6. Add regression coverage
   - Add tests for the onboarding welcome rendering helper: custom greeting + custom buttons are preserved with variables substituted.
   - Add tests for language persistence and no fallback to hardcoded welcome copy.
   - Add/adjust Concierge greeting tests so route/tone language defaults cannot overwrite a custom first-login welcome message.
